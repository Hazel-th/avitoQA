# Решение по API-тестам

## Что есть в решении

В задании нужны проверки четырёх основных ручек:

- создание объявления;
- получение объявления по `id`;
- получение всех объявлений продавца;
- получение статистики по `item id`.

Это всё покрыто автотестами.

Дополнительно я использую `DELETE /api/2/item/{id}`. Эта ручка помогает не оставлять мусор на стенде и держать тесты независимыми. Заодно через неё удобно проверить, что после удаления объявление действительно недоступно.

Если текст задания и примеры в коллекции расходятся, за основу брал именно Postman-коллекцию.

## Как разбит набор

- `smoke` — короткая проверка, что основные сценарии вообще работают;
- `regression` — основной набор с позитивными, негативными и граничными кейсами;
- `contract` — строгая проверка структуры ответов;
- `known_bugs` — кейсы на уже найденные дефекты стенда. Они помечены `xfail(strict=True)`.

Такой разбор нужен, чтобы можно было отдельно запускать стабильный набор и отдельно проверять уже известные баги.

## Как запустить

Установка зависимостей:

```bash
python3 -m venv .venv
./.venv/bin/pip install -r requirements.txt
```

Полный прогон:

```bash
./.venv/bin/python -m pytest -q
```

Только быстрые проверки:

```bash
./.venv/bin/python -m pytest -v -m smoke
```

Основной функциональный набор:

```bash
./.venv/bin/python -m pytest -v -m regression
```

Строгая проверка структуры ответов:

```bash
./.venv/bin/python -m pytest -v -m contract
```

Только кейсы на известные баги:

```bash
./.venv/bin/python -m pytest -v -m known_bugs
```

Один файл:

```bash
./.venv/bin/python -m pytest -v test_api.py
```

Один тест:

```bash
./.venv/bin/python -m pytest -v test_api.py -k "negative_price"
```

## Переменные окружения

По умолчанию используются такие значения:

- `BASE_URL=https://qa-internship.avito.com`
- `API_TIMEOUT_SECONDS=15`
- `SELLER_FIELD_MODE=auto`
- `TEST_RUN_ID=<любой идентификатор прогона>`

Примеры:

```bash
BASE_URL=https://qa-internship.avito.com ./.venv/bin/python -m pytest -v
```

```bash
TEST_RUN_ID=local-check ./.venv/bin/python -m pytest -v
```

### `SELLER_FIELD_MODE`

В документации встречаются оба варианта поля продавца: `sellerID` и `sellerId`.

Поддержаны три режима:

- `auto` — сначала попробовать определить рабочий вариант автоматически;
- `sellerID` — использовать только `sellerID`;
- `sellerId` — использовать только `sellerId`.

По умолчанию оставлен `auto`.

## Что важно в реализации

- Каждый тест создаёт свои данные и не зависит от порядка запуска.
- `sellerId` и имя объявления строятся из `TEST_RUN_ID` и `pytest nodeid`, чтобы прогон можно было проще воспроизвести.
- Всё, что создалось во время теста, потом удаляется.
- Во всех проверках используются обычные `assert`.
- Поведение живого стенда местами отличается от коллекции, поэтому в коде отдельно разделены функциональные проверки и строгие проверки контракта.
- Повторяющиеся входные данные для негативных кейсов вынесены в `test_data.py`.

## Как понимать результат прогона

Если сервис работает по контракту и без дефектов, все тесты должны быть зелёными.

На стенде `https://qa-internship.avito.com` прогон от **20 апреля 2026** дал такой результат:

- `39 passed`
- `7 xfailed`

`xfailed` здесь не случайные падения, а воспроизведение багов из `BUGS.md`.
